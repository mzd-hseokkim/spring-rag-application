## 🔥 임무 1: 의미 겹침 감지 및 관점 분리

outline에는 **제목이 비슷하거나 의미가 겹치는 leaf**가 흔히 있습니다.
이런 leaf 쌍에는 각각 **완전히 다른 topics**를 할당해야 합니다.

### 관점 매트릭스 (Perspective Matrix)

같은 주제 영역을 다루는 두 섹션이라도 서로 다른 관점을 적용하세요:

| 관점 코드 | 질문 | 어떤 종류의 topic이 해당? |
|---|---|---|
| **WHAT** | 사용자/이해관계자가 무엇을 할 수 있나? | 기능, 사용자 경험, 외부 인터페이스 |
| **HOW-tech** | 시스템 내부에서 어떻게 구현되나? | 아키텍처, 알고리즘, 인프라, 데이터 구조 |
| **WHY** | 왜·무엇을 목표로? | 비전, 전략, 차별화, 기대효과, KPI |
| **HOW-method** | 어떤 절차/방법론으로 진행하나? | 방법론, 일정, 마일스톤, 단계별 절차 |
| **OPS** | 어떻게 운영·유지되나? | 모니터링, 장애 대응, SLA, 유지보수 |
| **MGMT** | 어떻게 통제·조정되나? | 위험·이슈·변경 관리, 거버넌스 |
| **CTRL-tech** | 기술적 통제 (구현하는 것) | 암호화, 접근 제어, 인증, 입력 검증 |
| **CTRL-mgmt** | 관리적 통제 (운영하는 것) | 보안 교육, 서약서, 감사, 사고 대응 절차 |

### 적용 절차

1. **두 leaf의 title이 의미상 겹치는지 검사**
   - title 텍스트 또는 상위 경로가 같은 주제 영역(예: 기술/관리/전략/품질/보안 등)에 속하면 겹침
2. **겹치면 매트릭스에서 서로 다른 관점 코드 부여**
   - 한쪽은 WHAT, 다른 쪽은 HOW-tech (또는 WHY vs HOW-method 등)
3. **각 관점에 맞는 topics만 해당 leaf에 할당**
   - 한쪽 관점의 topic이 다른 쪽 leaf에 들어가면 위반

### 추상 예시

가상의 outline:
- `Section.A` (title: "주요 기능") — WHAT 관점 적용
- `Section.B` (title: "적용 기술 / 구현 방식") — HOW-tech 관점 적용

같은 도메인 주제(예: "사용자 조회 처리")가 둘에 모두 관련될 수 있지만:
- ✅ `Section.A.topic`: "사용자 조회 화면 및 입력 검증" (WHAT)
- ✅ `Section.B.topic`: "조회 쿼리 옵티마이저 및 캐시 계층 설계" (HOW-tech)
- ❌ 두 leaf 모두에 "사용자 조회 처리"라는 동일 topic 금지

또 다른 가상 예시:
- `Section.X` (title: "추진 전략") — WHY 관점
- `Section.Y` (title: "추진 방법론") — HOW-method 관점

- ✅ `Section.X.topics`: ["사업 비전", "핵심 성공 요인", "차별화 포인트"]
- ✅ `Section.Y.topics`: ["개발 방법론 선정", "단계별 일정", "변경 관리 절차"]
- ❌ "단계별 일정"이 두 leaf 모두에 등장 금지

### Cross-cutting 주제 처리

아래와 같은 주제는 여러 leaf에 자연스럽게 걸칠 수 있지만, **반드시 하나의 leaf에만** 배치하세요:
- "성능", "응답시간", "처리량" 류 → 하나의 leaf
- "보안", "암호화", "접근제어" 류 → 가장 적합한 보안/기술 leaf 하나
- "데이터 품질", "검증" 류 → 하나의 leaf
- "AI 안전성", "환각 방지", "가드레일" 류 → 하나의 leaf

여러 leaf에 동일 cross-cutting 주제가 나타나면 위반입니다.

## 🔥 임무 2: 요구사항 기반 topics 작성

요구사항 목록의 각 항목은 고유 ID(예: SFR-001, NFR-001, IFR-001 등 RFP에 따라 prefix 다양)와 item 이름을 가집니다.

### 절대 규칙

1. **기술/기능 성격 leaf**의 topics는 요구사항 item 이름을 **verbatim으로** 사용
   - topic 형식: `"{item 이름} ({REQ-ID})"` 또는 `"{item 이름} ({REQ-ID1}, {REQ-ID2})"`
   - item 이름은 요구사항 목록에서 **그대로 복사** (paraphrase·추상화 금지)

2. **요구사항 없는 leaf** (사업 개요, 전략, 일정 등 추상 섹션)는 추상 topics 사용
   - REQ-ID 불필요

3. **각 REQ-ID는 정확히 한 leaf의 topics에만 등장** (전역 uniqueness)
   - 같은 REQ가 두 leaf에 나타나면 위반
   - 충돌 시 임무 1의 관점 매트릭스로 한쪽 선택

4. **모든 요구사항이 어떤 leaf에는 반드시 할당**
   - 누락 최소화. 받은 N개 요구사항 → 모든 N개가 전체 topics 어딘가에 등장

### 추상 예시

좋은 topic:
- ✅ `"{item 이름 그대로} (SFR-001)"`
- ✅ `"{묶인 item 이름} (NFR-002, NFR-003)"`

나쁜 topic:
- ❌ `"기능 요약"` (item 이름 아님, 추상화)
- ❌ `"성능 최적화"` (item 이름 없음, 모호)

### 그룹핑 (관련 REQ 묶기)

관련된 요구사항 2~5개를 하나의 topic으로 묶어도 됩니다. 묶을 때 모든 ID를 괄호에 나열:
- ✅ `"{공통 주제 표현} (REQ-A, REQ-B, REQ-C)"`

## 🔥 임무 3: topics 개수 = 배점 비례

### 추상 leaf (요구사항 매칭 없음)
배점 비율에 따라:
- 총점의 15% 이상: **5~7개**
- 총점의 8~14%: **3~5개**
- 총점의 1~7%: **2~3개**

### 기술 leaf (요구사항 매칭 있음)
매칭된 요구사항 수를 기반으로 결정하되, 저배점 leaf는 그룹핑으로 세분화 억제:

- **저배점(5%↓)**: 관련 REQ를 그룹핑하여 최종 **2~3개** topics
  예: 매칭 REQ 8개 → topics 2~3개로 묶기
- **중배점(5~10%)**: 그룹핑하여 최종 **3~5개** topics
- **고배점(10%↑)**: 약한 그룹핑, 최종 **5~10개** topics

**원칙**: 배점이 낮은 섹션이 배점이 높은 섹션보다 많은 topics를 가지면 안 됩니다 (가중치 역전 금지).

저배점 그룹핑 예시 (추상):
- ❌ 8개 separate: `["X1 (REQ-1)", "X2 (REQ-2)", ..., "X8 (REQ-8)"]`
- ✅ 3개 grouped: `["X 군 1 (REQ-1, REQ-2, REQ-3)", "X 군 2 (REQ-4, REQ-5)", "X 군 3 (REQ-6, REQ-7, REQ-8)"]`

## 🔥 임무 4: 배점 할당 (weight)

각 leaf에 weight(정수)를 할당:
1. 평가 배점표의 항목을 가장 의미가 가까운 leaf에 매칭
2. 한 평가항목이 여러 leaf에 걸치면 의미 비중에 따라 분할
   예: "X 영역 40점"이 leaf A 25점 + leaf B 8점 + leaf C 7점
3. 매칭 없으면 `null`
4. 모든 leaf의 weight 합이 totalScore에 근접

저배점 leaf가 고배점 leaf와 같은 weight를 받으면 잘못된 것입니다.

---

## 🔥 임무 5: 의무 작성 항목 할당

각 MandatoryItem을 **정확히 한 leaf**에 배치:
- 모든 의무 항목이 어떤 leaf에는 배치 (누락 금지)
- 한 항목이 두 leaf에 배치되면 안 됨

## 🔥 임무 6: "기타 / 미분류" 섹션의 특별 규칙

대부분의 outline에는 "기타사항", "기타 고려사항", 마지막 chapter 등 **bucket 역할의 leaf**가 있습니다.
이 leaf는 **다른 chapter의 어느 leaf에도 자연스럽게 들어가지 않는 항목만** 담는 곳이지, 이미 배치된 항목을 재나열하는 공간이 아닙니다.

### 절대 금지

- ❌ "기타" 섹션에 다른 chapter의 leaf 제목을 그대로 또는 약간 변형하여 재나열
- ❌ 평가 카테고리 라벨(예: "평가항목 1-1") 자체를 topic으로 만들기 (배점표에 이미 있음)
- ❌ 다른 chapter에서 이미 다룬 cross-cutting 주제(보안, 성능, 품질 등)를 재나열

### 올바른 사용

- ✅ 어느 chapter의 leaf에도 의미상 맞지 않는 **고유 항목만** 배치
- ✅ 프로젝트 운영성 항목(요구사항 관리, 산출물 관리, 검수 절차 등)이 다른 곳에 없으면 여기에
- ✅ RFP의 "기타 고려사항", "특수 요건" 등에만 등장하는 항목

### 판단 기준

각 후보 topic에 대해 자문하세요: **"이 topic이 다른 어떤 leaf에 더 잘 맞나?"**
- 더 잘 맞는 leaf가 있으면 → 그쪽에 배치
- 어디에도 맞지 않으면 → 기타 leaf에 배치

기타 leaf는 **default empty** 상태가 자연스러우며, 채울 항목이 없으면 빈 채로 두는 것이 더 낫습니다.

## 응답 전 self-check (반드시!)

응답 전에 스스로 검증하세요:
1. ✅ **모든 leaf key가 출력에 포함됐는가?**
2. ✅ **REQ-ID 중복 검증**: 모든 topic에서 괄호 내 REQ-ID 추출 → 중복이 있으면 한쪽 제거
3. ✅ **요구사항 매칭이 가능한 leaf의 topics에 ID가 포함됐는가?** (REQ-ID 없는 추상 topic만 있으면 안 됨)
4. ✅ **모든 입력 REQ-ID가 전체 topics에 한 번씩은 나타나는가?** (누락 최소화)
5. ✅ **의미상 겹치는 leaf 쌍에 관점 매트릭스를 적용했는가?**
6. ✅ **저배점 leaf에 과도한 topics(6개+)가 없는가?** 있으면 그룹핑으로 축소
7. ✅ **저배점 leaf의 topics 수 < 고배점 leaf의 topics 수** (역전 없음)
8. ✅ **"기타" 섹션이 다른 chapter의 leaf 제목을 재나열하지 않는가?**
9. ✅ **평가 카테고리 라벨이 topics에 들어가지 않았는가?**
10. ✅ **모든 MandatoryItem이 정확히 한 leaf에 배치됐는가?**

## 출력 형식

반드시 아래 JSON 객체 형식으로만 응답하세요 (JSON 외 다른 텍스트·설명 없이).
모든 leaf의 key가 반드시 포함되어야 합니다.

```
{{
  "<leafKey1>": {{
    "weight": <정수 또는 null>,
    "topics": ["<topic 문자열>", "..."],
    "mandatoryItemIds": ["<MAND-ID>", "..."]
  }},
  "<leafKey2>": {{
    "weight": <정수 또는 null>,
    "topics": ["..."],
    "mandatoryItemIds": []
  }}
}}
```

### 추상 출력 예시 (도메인 무관)

```
{{
  "1.1": {{
    "weight": 10,
    "topics": ["배경 및 목적", "추진 근거", "기대 효과"],
    "mandatoryItemIds": []
  }},
  "2.1": {{
    "weight": 25,
    "topics": [
      "{item 이름 그대로} (SFR-001)",
      "{item 이름 그대로} (SFR-002, SFR-003)",
      "{item 이름 그대로} (IFR-001)"
    ],
    "mandatoryItemIds": ["MAND-01"]
  }},
  "2.2": {{
    "weight": 25,
    "topics": [
      "{2.1과는 다른 관점의 item 이름 그대로} (NFR-001)",
      "{2.1과는 다른 관점의 item 이름 그대로} (NFR-002, NFR-003)"
    ],
    "mandatoryItemIds": []
  }}
}}
```

위 예시에서 `2.1`과 `2.2`가 의미상 겹치는 sibling이라면 (예: 한쪽은 기능, 다른 쪽은 적용 기술), 관점 매트릭스로 분리하여 **서로 다른 topics**를 가져야 합니다.
