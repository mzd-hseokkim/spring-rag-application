## 분석 절차 (이 순서대로 사고하되, 최종 JSON만 출력하세요)

### 제안서가 제공된 경우 (proposalProvided = true)
갭 매트릭스 기반으로 질문을 생성합니다:
1. 갭 매트릭스에서 중요도 "상"이면서 "누락"/"미흡" 상태인 항목을 질문 소재 1순위로 선정
2. 중요도 "상"이면서 "부분충족" 항목을 2순위로 선정
3. 해당 페르소나의 관심 분야에 해당하는 갭 항목을 우선 선정
4. 각 질문은 갭 매트릭스의 특정 요구사항(REQ-XX)을 근거로 생성
5. 답변 작성 시 고객 문서, 제안 문서, 참조 문서, 웹 검색 등 모든 가용 소스를 활용
6. 각 질문/답변의 출처를 정확히 구분하여 sources에 기록

### 제안서가 제공되지 않은 경우 (proposalProvided = false)
제안서가 아직 작성되지 않은 상황이므로, "누락" 여부가 아닌 **요구사항 자체의 핵심 쟁점**에 집중합니다:
1. 중요도 "상" 요구사항부터 우선 선정
2. 해당 페르소나의 관심 분야에 해당하는 요구사항을 우선 선정
3. 각 요구사항에 대해 평가위원이 실제로 물어볼 법한 **구체적이고 실무적인 질문** 생성
   - 예: 구현 방안, 기술 선택 근거, 일정/리소스 타당성, 리스크 대응, 성능 기준 충족 방법 등
4. "~~이 누락되었습니다" 또는 "~~에 대한 내용이 없습니다" 같은 갭 지적 질문은 생성하지 않음
5. 답변은 해당 요구사항에 효과적으로 대응하기 위해 포함해야 할 핵심 포인트를 정리
6. sources에는 "고객문서:" 출처와, 참조/웹 자료가 있으면 해당 출처를 기록

## 평가 영역 카테고리 (category에 아래 중 하나를 사용하세요)
- 사업이해도: 사업 목적/배경에 대한 이해, 요구사항 분석의 정확성
- 기술적합성: 제안 기술의 적절성, 아키텍처, 기술적 실현 가능성
- 수행방안: 구체적 수행 절차, 방법론, 단계별 산출물
- 프로젝트관리: 일정, 위험관리, 품질관리, 의사소통 체계
- 인력/실적: 투입인력 전문성, 유사사업 수행 경험
- 지원/유지보수: 하자보수, 기술이전, 교육훈련 계획

## 난이도 기준
- 하(사실 확인): 문서에 명시된 내용을 확인하는 수준 — 최대 20%
- 중(분석/적용): 실행 가능성, 논리적 일관성을 따지는 수준 — 약 50%
- 상(평가/판단): 대안 비교, 리스크 예측, 전략적 판단 — 약 30%

## 출처(sources) 작성 규칙
각 질문과 답변의 근거 출처를 다음 형식으로 기록하세요:
- 고객 문서 근거: "고객문서: 섹션명 또는 요구항목"
- 제안 문서 근거: "제안문서: 섹션명 또는 페이지"
- 참조 문서 근거: "참조문서: 문서명/섹션명"
- 웹 검색 근거: "웹: 제목 또는 URL"
- 문서에 누락된 사항: "누락: 해당 항목이 제안서에 없음"

## 피해야 할 답변 스타일
- "~해야 합니다", "~하세요" 같은 지시/권유형 (X)
- "~에 대해 설명드리겠습니다" 같은 구어체/대화체 (X)
→ 대신: "~을 언급", "~수치 근거 제시", "~사례 포함" 등 핵심 포인트 나열
