## ⚠️ 추출 대상

### ✅ 추출 대상
- 고유 ID를 가진 **요구사항 항목**: RFP에 명시된 prefix(예: SFR-, NFR-, IFR-, PFR-, DAR-, PER-, PMR-, SER-, COR-, QUR-, PSR-, TER-, INR-, SIR- 등 RFP마다 다양) + 일련번호
- ID가 없어도 "~해야 한다", "~을 구축한다", "~을 제공한다" 형태의 **구체적 요구사항**
- 시스템·서비스·운영이 충족해야 할 기능, 성능, 품질, 보안, 인터페이스, 데이터, 관리 항목

### ❌ 추출 금지 (중요!)
- **평가항목 / 평가기준표 / 배점표는 여기서 추출하지 마세요.** 별도의 RFP mandates 추출 단계에서 처리됩니다.
- "평가항목 1-1", "평가항목 1-2" 같은 **평가 카테고리 라벨**을 요구사항으로 복사하지 마세요.
- 예: "사업 이해 (10점)", "기술 적합성 (40점)" 같은 평가 항목 라벨은 배점표이지 요구사항이 아닙니다.
- 평가 카테고리는 outline의 chapter/section으로 이미 다뤄지므로, 여기서 또 추출하면 중복 데이터가 됩니다.

**판단 기준**: "이것이 시스템이 충족해야 할 기능·성능·품질인가?" 라고 자문하세요. "평가위원이 제안서를 평가하는 기준"이라면 추출 금지.

## 추출 기준
- 각 항목에 **중요도**를 부여: 필수/핵심 요건이면 "상", 일반 요건이면 "중", 선택/부가 요건이면 "하"
- **정량 배점이 요구사항 항목에 직접 명시되어 있으면** `weight` 필드에 정수로 기록 (드문 경우)
  - 배점이 없으면 `weight`를 `null`로 두세요
- **누락 금지**: RFP 본문에서 발견된 모든 고유 ID 요구사항은 반드시 포함
- 통상 RFP에 따라 수십~수백 개의 요구사항이 있을 수 있음. 모두 포함

## 카테고리 (category 필드에 사용)
RFP의 주제 영역에 따라 자유롭게 부여하되, 가능하면 RFP 본문의 분류 체계를 따르세요. 일반적인 카테고리 예:
- 기능 (Functional)
- 성능 (Performance)
- 인터페이스 (Interface)
- 데이터 (Data)
- 보안 (Security)
- 품질 (Quality)
- 관리 (Management)
- 운영 (Operation)
- 제약 (Constraint)
- 기타 (Other)

위 목록은 가이드일 뿐이며, RFP에 다른 분류가 있으면 그것을 사용하세요.

## ID 부여 규칙
- **RFP 본문에 명시된 고유 요구사항 번호는 그대로 사용** (prefix 형식 무관)
  예: SFR-001, NFR-12, IFR_3.2.1, REQ-001 등 RFP에 나타난 형식 그대로
- **단, "평가항목 1-1" 같은 평가 카테고리 번호는 절대 id로 사용 금지** (요구사항이 아닌 배점표)
- 본문에 번호가 없는 항목만 "REQ-01", "REQ-02" 형식으로 자동 부여

## 출력 형식
반드시 아래 JSON 배열 형식으로만 응답하세요 (JSON 외 다른 텍스트 없이).
`weight`는 정량 배점이 명시되어 있으면 정수, 없으면 `null`로 두세요.

```
[
  {{
    "id": "<RFP 원본 ID 또는 REQ-NN>",
    "category": "<카테고리>",
    "item": "<요구사항 item 이름 — RFP 본문 그대로>",
    "description": "<요구사항 상세 설명>",
    "importance": "<상|중|하>",
    "weight": <정수 또는 null>
  }}
]
```
