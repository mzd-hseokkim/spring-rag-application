## 🔥 가장 중요한 원칙 — 카테고리 폭주 방지

**한 leaf에 너무 많은 요구사항이 몰리면 안 됩니다.** 일반적인 leaf는 **5~10개의 children**이 적절합니다.
이를 위해 **카테고리의 항목 수**를 보고 분배 결정을 다르게 해야 합니다:

| 카테고리 항목 수 | 매핑 전략 | 예시 |
|---|---|---|
| **1~7개** | **1:1 매핑** — 가장 적합한 leaf 1개에 모두 배치 | "성능: 5개" → ["3.3"] |
| **8~15개** | **1:2 매핑** — 의미가 가까운 leaf 2개에 분산 | "보안: 12개" → ["4.3", "5.3"] |
| **16~30개** | **1:2~3 매핑** — 2~3개 leaf에 분산 | "관리: 23개" → ["5.1", "5.2", "5.4"] |
| **30개 초과** | **1:3+ 매핑** — 3개 이상 leaf에 분산 | "기능: 34개" → ["3.1", "4.1"] (둘 다 기술 성격) |

**절대 한 leaf에 30개+가 가도록 매핑하지 마세요.** 만약 30개+ 카테고리에 1:1 매핑하면 그 leaf는 폭주합니다.

## 🎯 임무

### 임무 1: 카테고리 → leaf 매핑 (1:N 적극 활용)

각 카테고리에 대해:
1. 카테고리의 항목 수를 확인
2. 위 표의 매핑 전략에 따라 1개 또는 여러 leaf 선택
3. 의미적으로 가까운 leaf를 선택 (leaf title + 상위 경로 참고)
4. 같은 카테고리를 분산할 때는 **관점이 다른 leaf** (예: WHAT vs HOW-tech) 또는 **내용 영역이 다른 leaf** (예: 인증 vs 권한)를 선택

**핵심 규칙**:
- 항목이 **많을수록 더 많은 leaf에 분산**
- 분산 시 **의미상 어색하지 않은 leaf만** 선택 (관련성 있는 leaf만)
- 무관한 leaf에 강제 매핑 금지
- **전용 leaf 우선 + WHAT↔HOW-tech 분배 허용**:
  - 카테고리와 의미가 일치하는 전용 leaf(title에 해당 키워드 포함)가 있으면 **우선 매핑**
  - 단, 같은 카테고리를 **WHAT leaf("XX 요구사항")와 HOW-tech leaf("적용기술", "개발환경" 등) 양쪽에 분배하는 1:N 매핑은 허용하고 권장**합니다. 이것이 "무엇을 달성 vs 어떻게 구현"의 관점 분리를 만듭니다.
  - 예: "기능" 카테고리 → ["기능 요구사항"(WHAT), "적용기술"(HOW-tech)] 양쪽에 매핑 ✅
  - 예: "성능" 카테고리 → ["성능 요구사항"(WHAT), "적용기술"(HOW-tech)] 양쪽에 매핑 ✅
  - 예: "데이터" 카테고리 → ["데이터 요구사항"(WHAT), "개발환경"(HOW-tech)] 양쪽에 매핑 ✅
- **범용 leaf 보호 (변경 없음)**: "전략", "추진체계", "방법론", "기술지원" 같은 범용 leaf에는 전용 leaf가 이미 있는 카테고리를 매핑하지 마세요. 특히 항목 수가 많은 카테고리(보안 등)가 범용 leaf에 몰리면 해당 leaf가 해당 카테고리 전용이 되어 버립니다

### 임무 2: leaf의 관점 역할 부여 (1:N 분배의 가이드)

각 leaf에 관점 역할(role)을 부여합니다. 이 role은 1:N 분배 시 어떤 leaf로 분산할지 결정에 사용됩니다.

**관점 코드**:
| 코드 | 의미 | 적용 대상 |
|---|---|---|
| WHAT | 달성 목표·요구사항 | title에 **"요구사항"**이 포함된 섹션 (기능·성능·데이터·인터페이스·보안·테스트 요구사항 등). "**무엇을** 달성해야 하는가"를 다루는 섹션 |
| HOW-tech | 기술 구현·아키텍처 | title에 "적용기술", "개발환경", "아키텍처", "구현" 등이 포함된 섹션. "**어떻게** 구현하는가"를 다루는 섹션 |
| WHY | 전략·비전 | 전략·목표·차별화 관련 섹션 |
| HOW-method | 방법론·절차 | 방법론·일정·추진 관련 섹션 |
| OPS | 운영·유지보수 | 운영·유지보수·모니터링 관련 섹션 |
| MGMT | 관리·통제 | 프로젝트관리·위험관리·거버넌스 관련 섹션 |
| CTRL-tech | 기술적 통제 | 기술 구현으로서의 보안·품질 (암호화·SQL 인젝션 등) |
| CTRL-mgmt | 관리적 통제 | 프로세스로서의 보안·품질 (교육·서약서·감사 등) |
| MISC | 기타·미분류 | 다른 어느 role에도 해당 안 되는 섹션 |

**⚠️ WHAT vs HOW-tech 판단 핵심 규칙**:
- title에 "요구사항"이 포함되면 **반드시 WHAT**. "성능 요구사항", "데이터 요구사항", "인터페이스 요구사항" 등은 모두 "무엇을 달성해야 하는가"를 다루는 섹션이지 "어떻게 구현하는가"가 아닙니다.
- "적용기술", "개발환경", "수행기반" 같은 title만 HOW-tech.
- 헷갈리면: "이 섹션이 목표 수치와 달성 기준을 정의하는가(WHAT) vs 구현 수단을 기술하는가(HOW-tech)"로 판단.

**규칙**:
1. 각 leaf에 정확히 하나의 role 부여
2. 의미상 겹치는 sibling leaf는 **반드시 다른 role** 부여
3. 기타사항/미분류 섹션은 "MISC"

### 임무 3: 큰 카테고리의 1:N 분산 가이드

큰 카테고리(15개+)를 분산할 때는 다음 패턴을 따르세요:

**예 1 — "기능" 카테고리 34개**:
- 후보 leaf 탐색: 기능 관련 leaf들 (예: "기능 요구사항", "적용기술", "주요 기능")
- 의미상 가장 가까운 2~3개를 선택 (예: WHAT role + HOW-tech role)
- 결과: `"기능": ["3.1", "4.1"]` — RuleBasedPlanner가 round-robin으로 17개씩 분배

**예 2 — "보안" 카테고리 22개**:
- 후보 leaf 탐색: 보안 관련 leaf들 (예: "보안 요구사항", "기밀보안 관리")
- 한 leaf는 기술 관점(CTRL-tech), 다른 leaf는 관리 관점(CTRL-mgmt)
- 결과: `"보안": ["4.3", "5.3"]` — 11개씩 분배

**예 3 — "관리" 카테고리 30개**:
- 후보 leaf 탐색: 관리 관련 leaf들 (예: "일정관리", "품질관리", "위험관리", "기밀보안 관리")
- 3~4개에 분산 (각자 다른 sub-domain)
- 결과: `"관리": ["5.1", "5.2", "5.4"]` — 10개씩 분배

## 응답 전 self-check

응답 전에 스스로 검증하세요:
1. ✅ **모든 카테고리가 출력에 포함됐는가?**
2. ✅ **모든 leaf의 role이 부여됐는가?**
3. ✅ **항목 수가 30개 초과인 카테고리가 있다면 1개 leaf에만 매핑되지 않았는가?** (반드시 2개+ leaf에 분산)
4. ✅ **항목 수가 8개 이상인 카테고리는 가능하면 2개 이상 leaf에 분산했는가?**
5. ✅ **분산 대상 leaf가 의미적으로 합리적인가?** (무관한 leaf로 분산하지 않았는가)
6. ✅ **의미상 겹치는 sibling leaf에 서로 다른 role을 부여했는가?**

## 출력 형식
반드시 아래 JSON 객체 형식으로만 응답하세요 (JSON 외 다른 텍스트·설명 없이).

```
{{
  "categoryToLeafKeys": {{
    "<카테고리명>": ["<leafKey1>", "<leafKey2>", ...],
    ...
  }},
  "leafKeyToRole": {{
    "<leafKey1>": "<관점 코드>",
    "<leafKey2>": "<관점 코드>",
    ...
  }}
}}
```

### 추상 출력 예시 (도메인 무관)

가상의 RFP 카테고리 분포: 기능 34, 관리 23, 보안 22, 데이터 8, 성능 5, 인터페이스 3

```
{{
  "categoryToLeafKeys": {{
    "기능": ["3.1", "4.1"],
    "관리": ["5.1", "5.2", "5.4"],
    "보안": ["4.3", "5.3"],
    "데이터": ["3.2", "4.2"],
    "성능": ["3.3"],
    "인터페이스": ["3.4"]
  }},
  "leafKeyToRole": {{
    "1.1": "WHY",
    "2.1": "WHAT",
    "3.1": "WHAT",
    "3.2": "HOW-tech",
    "3.3": "HOW-tech",
    "3.4": "WHAT",
    "4.1": "HOW-tech",
    "4.2": "HOW-tech",
    "4.3": "CTRL-tech",
    "5.1": "MGMT",
    "5.2": "MGMT",
    "5.3": "CTRL-mgmt",
    "5.4": "MGMT"
  }}
}}
```

위 예시에서:
- **"기능" 34개**는 ["3.1", "4.1"]에 분산 → 각 leaf 약 17개
- **"관리" 23개**는 3개 leaf에 분산 → 각 leaf 약 8개
- **"보안" 22개**는 기술 관점(4.3)과 관리 관점(5.3)에 분산 → 각 leaf 11개
- **작은 카테고리(8개 이하)**도 가능하면 2개에 분산해 보았습니다 (데이터 8 → 2개)
- **5개 이하**는 1개로 충분 (성능 5, 인터페이스 3)
