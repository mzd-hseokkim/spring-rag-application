# Phase 1 — Core RAG Pipeline (MVP)

핵심 RAG 파이프라인을 구축하여 "문서 업로드 → 검색 → 응답 생성"의 end-to-end 흐름을 완성한다.

---

## 1. 문서 수집 및 처리 (Document Ingestion)

### 1-1. 문서 업로드 API
- REST API로 파일 업로드 수신 (PDF, TXT, Markdown)
- 업로드 즉시 202 Accepted 반환
- 문서 메타데이터(파일명, 타입, 크기, 업로드 일시) DB 저장

### 1-2. 비동기 처리 파이프라인
- 문서 파싱: 파일 타입별 텍스트 추출
- 시맨틱 청킹: 의미 단위 분할 (크기 + 오버랩 전략)
- 임베딩 생성: 청크별 벡터 임베딩 생성
- pgvector 저장: 임베딩 + 메타데이터 + 원본 텍스트 저장
- 처리 상태 관리: PENDING → PROCESSING → COMPLETED / FAILED

### 1-3. 처리 상태 알림
- SSE 엔드포인트로 문서 처리 진행 상태 실시간 전송

---

## 2. 하이브리드 검색 (Hybrid Retrieval)

### 2-1. 벡터 검색
- 사용자 질의 임베딩 → pgvector 유사도 검색 (cosine similarity)

### 2-2. 키워드 검색
- PostgreSQL tsvector/tsquery 기반 Full-Text Search
- 청크 저장 시 tsvector 컬럼 자동 생성

### 2-3. 하이브리드 퓨전
- 벡터 검색과 키워드 검색을 병렬 실행
- RRF (Reciprocal Rank Fusion)로 결과 병합
- 상위 K건 반환

---

## 3. 응답 생성 (Generation)

### 3-1. 컨텍스트 구성
- 검색된 청크들을 프롬프트 템플릿에 조합

### 3-2. LLM 스트리밍 응답
- Spring AI ChatClient를 통한 LLM 호출
- SSE 기반 토큰 단위 스트리밍 응답

### 3-3. 출처 표시
- 응답에 사용된 원본 청크의 문서 출처 정보 함께 반환

---

## 4. 채팅 UI (Frontend)

### 4-1. 기본 채팅 인터페이스
- 메시지 입력 및 전송
- SSE 스트리밍 응답 실시간 표시

### 4-2. 문서 업로드 UI
- 파일 드래그앤드롭 / 선택 업로드
- 처리 상태 실시간 표시

### 4-3. 출처 참조 UI
- 답변 하단 또는 인라인으로 참조 문서 정보 표시

---

## 처리 방식 요약

| 기능 | 동기/비동기 | 비고 |
|------|------------|------|
| 문서 업로드 수신 | 동기 | 202 Accepted 즉시 반환 |
| 문서 파싱 → 청킹 → 임베딩 | 비동기 | 백그라운드 파이프라인 |
| 처리 상태 알림 | SSE | 실시간 푸시 |
| 벡터 검색 ∥ 키워드 검색 | 동기, 병렬 | 두 검색 동시 실행 |
| 하이브리드 퓨전 (RRF) | 동기 | 검색 완료 후 즉시 |
| LLM 응답 생성 | SSE 스트리밍 | 토큰 단위 전송 |
