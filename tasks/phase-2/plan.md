# Phase 2 — Retrieval Enhancement + Conversation

검색 정밀도를 높이고, 멀티턴 대화와 시맨틱 캐싱으로 사용자 경험과 비용 효율을 개선한다.

---

## 1. 리랭킹 (Reranking)

### 1-1. Cross-Encoder 리랭킹
- 하이브리드 검색 결과 상위 20~50건을 Cross-encoder로 정밀 재정렬
- Spring AI의 DocumentPostProcessor API 활용
- 최종 상위 K건만 컨텍스트로 전달

> **처리 방식**: 동기 — 하이브리드 퓨전 직후, 응답 생성 전

---

## 2. 쿼리 변환 (Query Transformation)

### 2-1. Multi-Query Expansion
- 사용자 질의를 LLM으로 다수의 의미적 변형 질의로 확장
- 각 변형 질의로 검색 → 결과 통합
- Spring AI의 MultiQueryExpander 활용

### 2-2. 대화 압축 (Conversation Compression)
- 멀티턴 대화를 단일 독립 질의로 압축
- Spring AI의 CompressionQueryTransformer 활용
- 이전 대화 맥락 손실 없이 검색 정확도 확보

> **처리 방식**: 동기 — 검색 파이프라인 진입 전 전처리

---

## 3. 멀티턴 대화 관리 (Conversation)

### 3-1. 대화 이력 저장
- Redis에 세션별 메시지 이력 저장
- TTL 기반 자동 만료

### 3-2. 세션 관리
- 사용자별/세션별 대화 분리
- 새 세션 생성 / 이전 세션 이어가기

### 3-3. 대화 컨텍스트 윈도우
- 토큰 한도 초과 시 오래된 메시지 요약/압축

> **처리 방식**: 동기 — 요청 수신 시 이력 조회, 응답 후 이력 저장

---

## 4. 쿼리 라우팅 (Query Routing)

### 4-1. 의도 분류
- 질의 유형 판별 (사실 조회, 비교, 일반 대화 등)
- 임베딩 유사도 기반 또는 LLM 분류

### 4-2. 라우팅 분기
- RAG 대상 → 검색 파이프라인
- 일반 대화/인사 → LLM 직접 응답 (검색 생략)
- 구조화 데이터 질의 → SQL 조회 (확장 가능)

> **처리 방식**: 동기 — 검색 파이프라인 진입 전

---

## 5. 시맨틱 캐싱 (Semantic Cache)

### 5-1. 캐시 조회
- 사용자 질의를 임베딩 → Redis에서 유사 질의 검색
- 유사도 임계치 이상이면 캐시된 응답 즉시 반환

### 5-2. 캐시 저장
- 새 응답 생성 후 질의-응답 쌍을 Redis에 저장
- TTL 기반 캐시 만료

### 5-3. 캐시 무효화
- 문서 추가/삭제 시 관련 캐시 엔트리 무효화

> **처리 방식**: 조회는 동기 (파이프라인 최전방), 저장은 비동기 (응답 반환 후)

---

## 처리 방식 요약

| 기능 | 동기/비동기 | 비고 |
|------|------------|------|
| 시맨틱 캐시 조회 | 동기 | 파이프라인 최전방 |
| 대화 압축 | 동기 | 검색 전 전처리 |
| 의도 분류 + 라우팅 | 동기 | 검색 전 |
| Multi-Query Expansion | 동기 | 검색 전 |
| 리랭킹 | 동기 | 퓨전 직후 |
| 대화 이력 저장 | 동기 | 요청/응답 시 |
| 시맨틱 캐시 저장 | 비동기 | 응답 반환 후 |
