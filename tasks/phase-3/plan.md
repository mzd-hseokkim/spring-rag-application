# Phase 3 — LLMOps + Quality Assurance

RAG 파이프라인의 운영 가시성을 확보하고, 응답 품질을 체계적으로 관리한다.

---

## 1. 관측성 (Observability)

### 1-1. 파이프라인 트레이싱
- 요청 단위로 각 단계(검색, 리랭킹, 생성)의 latency 기록
- 단계별 입출력 데이터 구조화 로깅
- trace ID로 요청 전체 흐름 추적 가능

### 1-2. 토큰 사용량 추적
- 요청당 input/output 토큰 수 기록
- 모델별 단가 기반 비용 추정
- 응답 메타데이터에 토큰 정보 포함

### 1-3. 검색 품질 로깅
- 검색된 청크 ID, 유사도 점수, 리랭킹 점수 기록
- 캐시 히트/미스 비율 추적

> **처리 방식**: 비동기 — 응답과 병렬로 로그 적재 (요청 처리 latency에 영향 없음)

---

## 2. 평가 (Evaluation)

### 2-1. Faithfulness 평가
- 응답이 검색된 컨텍스트에 근거하는지 자동 검증 (LLM-as-judge)
- 환각 여부 탐지

### 2-2. Relevance 평가
- 검색 결과가 질의에 적합한지 평가 (Precision/Recall)
- 응답이 질의에 대한 적절한 답변인지 평가

### 2-3. Groundedness 검증
- 응답 내 각 주장이 특정 청크에 매핑되는지 사후 검증
- 검증 실패 시 로깅 (즉시 차단은 하지 않음)

### 2-4. 사용자 피드백 집계
- 👍/👎 피드백 데이터 수집 및 트렌드 분석
- 부정 피드백이 많은 질의 패턴 식별

> **처리 방식**: 비동기 — 배치 또는 샘플링 기반 (전수 평가는 비용 과다)

---

## 3. 프롬프트 관리 (Prompt Management)

### 3-1. 프롬프트 버전 관리
- 시스템 프롬프트 / RAG 프롬프트 템플릿의 버전 추적
- DB 또는 파일 기반 버전 관리
- 어떤 버전이 어떤 요청에 사용되었는지 기록

### 3-2. A/B 테스트
- 프롬프트 변형별 트래픽 분배
- 평가 메트릭과 연동하여 품질 비교

---

## 4. 비용 및 가드레일

### 4-1. Rate Limiting
- 사용자/세션별 요청 빈도 제한
- Redis 기반 sliding window 카운터

### 4-2. 입력 검증
- 프롬프트 인젝션 패턴 필터링
- 최대 입력 길이 제한
- 민감 정보 마스킹

### 4-3. 비용 대시보드
- 일별/월별 API 호출 횟수 및 토큰 비용 집계
- 모델별/기능별 비용 분류

> **처리 방식**: Rate Limiting과 입력 검증은 동기 (요청 진입 시), 비용 집계는 비동기 (배치)

---

## 처리 방식 요약

| 기능 | 동기/비동기 | 비고 |
|------|------------|------|
| Rate Limiting | 동기 | Redis 기반, 요청 진입 시 |
| 입력 검증 | 동기 | 파이프라인 진입 전 |
| 파이프라인 트레이싱 | 비동기 | 응답과 병렬 |
| 토큰/비용 추적 | 비동기 | 응답 메타데이터 + 로그 |
| Faithfulness/Relevance 평가 | 비동기 | 배치 또는 샘플링 |
| Groundedness 검증 | 비동기 | 응답 후 백그라운드 |
| 사용자 피드백 집계 | 비동기 | 배치 집계 |
| 비용 대시보드 | 비동기 | 배치 집계 |
