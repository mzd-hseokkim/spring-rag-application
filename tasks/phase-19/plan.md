# Phase 19 — 문서 생성 품질 개선 (Research-Grounded Generation)

Phase 18에서 구축한 문서 생성 파이프라인의 품질을 근본적으로 개선한다.
현재 AI가 내용을 "지어내는" 방식에서, **사실에 근거하여 정리/편집하는** 방식으로 전환한다.

---

## 핵심 원칙

**AI는 작가가 아니라 편집자다.**

- AI가 창작하면 그럴듯하지만 거짓인 내용이 생성됨 (제안서에서는 치명적)
- AI는 수집된 자료를 기반으로 구조화/정리/요약하는 역할만 담당
- 참고 자료에 없는 내용은 작성하지 않도록 프롬프트에서 강제

---

## 현재 vs 개선 워크플로우

### 현재 (Phase 18)
```
사용자 입력 → 아웃라인 생성 → (얕은 RAG) → AI가 내용 생성 → 렌더링
```

### 개선 (Phase 19)
```
사용자 입력
    ↓
1. OUTLINE — AI가 목차/아웃라인 생성
    ↓
2. USER REVIEW — 사용자가 아웃라인 편집/확인 (새 단계)
    ↓
3. RESEARCH — 섹션별 심층 리서치 (새 단계)
   ├─ RAG 검색 (기존 문서, 다중 쿼리)
   ├─ 웹 검색 (선택적, 공개 정보 수집)
   └─ 수집 자료 정리 + 출처 기록
    ↓
4. DRAFT — 수집 자료 기반으로만 초안 작성
   └─ "제공된 자료에 없는 내용은 '[정보 필요]'로 표시"
    ↓
5. USER EDIT — 사용자가 초안 검토/수정 (선택적)
    ↓
6. RENDER — 최종 문서 렌더링
```

---

## 세부 계획

### 1. 아웃라인 사용자 편집 (OUTLINE → USER REVIEW)

- 아웃라인 생성 후 바로 섹션 생성으로 넘어가지 않음
- 프론트엔드에서 아웃라인(섹션 목록) 편집 UI 제공
  - 섹션 추가/삭제/순서 변경
  - 섹션별 keyPoints 편집
  - 사용자 확인 후 "생성 진행" 클릭
- 백엔드: GenerationJob에 `OUTLINE_REVIEW` 상태 추가
- API: `PUT /api/generations/{id}/outline` — 편집된 아웃라인 저장 + 진행

### 2. 섹션별 심층 리서치 (RESEARCH 단계)

현재 `SearchService.search(query)` 1회 호출로는 충분한 컨텍스트를 확보할 수 없다.

#### 2-1. 다중 쿼리 RAG 검색
- 섹션의 heading, purpose, keyPoints 각각으로 별도 검색
- 검색 결과를 합치고 중복 제거
- 기존 `SearchService`의 query expansion 기능 활용

#### 2-2. 웹 검색 연동 (선택적)
- 외부 검색 API (Tavily, SerpAPI, 또는 자체 크롤링)
- 사용자가 "웹 검색 포함" 옵션을 선택한 경우에만 활성화
- 검색 결과는 요약하여 리서치 컨텍스트에 추가
- 출처 URL 기록

#### 2-3. 리서치 결과 저장
- GenerationJob에 `researchData` (JSONB) 필드 추가
- 섹션별 수집 자료 + 출처 목록 저장
- 리서치 결과를 프론트엔드에서 확인 가능

### 3. 사실 기반 초안 작성 (DRAFT 단계)

#### 3-1. 프롬프트 개선
```
당신은 수집된 자료를 정리하는 편집자입니다.

규칙:
- 반드시 [참고 자료]에 포함된 내용만 사용하세요.
- 참고 자료에 없는 내용은 "[정보 필요: OO에 대한 데이터]"로 표시하세요.
- 모든 주요 주장에는 출처 번호를 [1], [2] 형태로 표기하세요.
- 추측이나 일반론을 작성하지 마세요.
```

#### 3-2. 출처 추적
- 생성된 content에서 참조된 출처를 references에 자동 매핑
- RAG 검색 결과의 문서 ID/파일명을 출처로 활용
- "[정보 필요]" 태그가 포함된 섹션은 사용자에게 알림

### 4. 사용자 편집 기능 (USER EDIT)

- 생성된 초안을 프론트엔드에서 섹션별로 편집 가능
- 마크다운 에디터 (간단한 textarea 또는 마크다운 에디터 라이브러리)
- "[정보 필요]" 부분을 하이라이트하여 사용자가 직접 보완
- 편집 완료 후 최종 렌더링 진행
- API: `PUT /api/generations/{id}/sections` — 편집된 섹션 저장

### 5. 품질 검증 자동화

- AI 리뷰 단계에서 각 섹션의 "근거 비율" 측정
  - 참고 자료에서 뒷받침되는 내용 비율
  - "[정보 필요]" 태그 수
- 품질 점수를 프론트엔드에 표시
- 품질 미달 섹션은 리서치 보강 후 재생성 가능

## 기술 변경 사항

### 백엔드
- `GenerationStatus`에 `OUTLINE_REVIEW`, `RESEARCHING`, `DRAFT_REVIEW` 추가
- `GenerationJob`에 `researchData` JSONB 컬럼 추가
- `ResearchService` 신규 — 다중 쿼리 RAG + 웹 검색 오케스트레이션
- `ContentGeneratorService` 프롬프트 전면 개편
- 아웃라인/섹션 편집 API 추가

### 프론트엔드
- `GeneratePage` 3단계 → 6단계 플로우 전환
- 아웃라인 편집 UI (드래그 앤 드롭 순서 변경, 섹션 추가/삭제)
- 리서치 결과 뷰어 (섹션별 수집 자료 확인)
- 섹션별 마크다운 에디터
- "[정보 필요]" 하이라이트 표시

### DB 마이그레이션
- `generation_job` 테이블에 `research_data JSONB` 컬럼 추가

---

## 구현 우선순위

1. **프롬프트 개편** — "편집자 모드" 프롬프트로 교체 (가장 빠르고 효과 큼)
2. **다중 쿼리 RAG** — 섹션별 심층 검색
3. **아웃라인 편집** — 사용자 확인 후 진행
4. **초안 편집** — 섹션별 사용자 수정
5. **웹 검색 연동** — 외부 정보 수집
6. **품질 검증** — 근거 비율 측정 + 자동 리뷰
